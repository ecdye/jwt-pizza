# Incident: 2026-04-01 11-00-00 MDT

## Summary

Between 11:00 MDT and 11:25 MDT (17:00 UTC to 17:25 UTC) on April 1, 2026, `POST /api/order` repeatedly failed with HTTP 500 while attempting to send newly created orders to the pizza factory. The failures were triggered by factory-side responses labeled `chaos monkey`, which indicates an upstream failure injection or outage at the factory rather than a malformed request generated by this service.

This was a medium-severity checkout incident because users could authenticate and submit orders, but fulfillment failed at the external factory dependency. Based on the provided log excerpt, at least six order attempts were affected. There is additional risk because our service persists the diner order locally before calling the factory, so users could see a checkout failure even though order records `1572` through `1577` were already created in our database.

## Detection

The incident was detectable from application logs beginning just after `2026-04-01 11:00 MDT`, where `POST /api/order` returned HTTP 500 with the message `Failed to fulfill order at factory`. Each 500 is paired with a `factory` log entry showing `chaos monkey`, which identifies the factory dependency as the failing component.

The provided evidence does not show an automated alert or a human page, so time-to-detection was limited by someone manually inspecting logs or following the support URL returned by the factory. We could cut detection time materially by alerting on consecutive `5xx` responses for `/api/order` and correlating them with non-OK responses from the factory integration.

## Impact

For 25 minutes between `17:00 UTC` and `17:25 UTC` on `04/01/26`, users attempting to place orders experienced checkout failures.

The provided log excerpt shows at least six failed order attempts and confirms at least one affected diner account, `d@jwt.com`. Externally, users would have seen HTTP 500 responses from `/api/order` and would not have received a valid factory JWT for verification. Internally, order rows were still inserted locally before the factory call failed, creating a risk of duplicate or orphaned orders if users retried.

At least one support workflow was involved, as every error response included the same factory support report URL. The excerpt does not contain evidence for a higher support case count.

## Response

No explicit on-call response, page, or operator action is captured in the provided log excerpt.

The observable application response was to return HTTP 500 to the caller and include the upstream `followLinkToEndChaos` support URL in the response body. The main obstacle to fast response is that the service depends directly on the factory result and appears to lack dedicated alerting for sustained factory failures.

## Root cause

A factory-side chaos experiment or failure mode caused the upstream pizza factory to reject order fulfillment requests with a `chaos monkey` response. Our service treats any non-OK factory response as a hard checkout failure after the local order has already been inserted, and it has no rollback, retry, or graceful degradation path for this dependency failure.

## Resolution

The incident was resolved by 11:25 MDT (17:25 UTC), when the factory-side chaos behavior stopped affecting order fulfillment and checkout requests were able to complete successfully again.

Based on the failure mode, restoration occurred once the factory resumed returning successful `2xx` responses for `/api/order`. Recovery should be confirmed by observing successful order submissions, a return of factory JWTs, and the disappearance of repeated `chaos monkey` factory log entries.

## Prevention

This class of incident can recur any time the factory dependency returns errors during checkout, because order creation and factory fulfillment are not coordinated transactionally. Even if the exact `chaos monkey` trigger is unique to this event, any upstream outage, auth issue, or malformed factory response would create the same user-visible failure and the same orphaned-order risk.

The missing mitigations are clear: staged chaos testing instead of production-impacting chaos, alerting on factory failure rates, and a safer local order state model such as `pending_fulfillment` rather than immediately treating the order as complete.
